Panels for dragging and dropping:

Many Acknex users asked us how could they do a panel with objects for an inventory. It seemed to be a difficult thing. Now the fully - commented script is here. Enjoy it!

First of all, we need a mouse pointer. This is done in mouse.wdl. We have decided to let the player activate the cursor when he needs, by right clicking; this way the mouse pointer won't stay in our way if we don't need it. We have placed the line

IF_RIGHT mouse_start;

at the end of move.wdl; we don't want to waste an each_tick action. We also have

SET IF_LEFT, mouse_check;

in move.wdl; this action checks if the mouse pointer is visible or not, and was added for those of you who intend to do an action / adventure game, because if the mouse pointer isn't visible, you should be able to shoot with the left button.

This panel has two skills: health and IQ; they can be increased by eating and finding keys. The objects that are to be displayed in our panel are contained in a font.

FONT objfont,<objfont.pcx>,28,22;

As you can see, it is a font that has 28x22 sized "letters". Objfont.pcx is a bitmap that contains 11 (0…9 and space) characters. Nobody can tell us to put numbers in, so we put pictures instead. In objfont.pcx we only use 0,1,2,3,4 for empty, greenkey, cake, fish, bluekey. This way, when we display 0, an empty space will appear, 1 = greenkey, 2 = cake and so on. So if I want to display the fish, I just say display 3? Right.

The big difference between overlays and panels is that overlays can't contain digits. So we're going to use a panel as our big panel:

PANEL big_panel {
PAN_MAP bigpanel_map;
POS_X 0;
POS_Y 355; // the panel has 320x45 pixels; if we choose POS_Y 355
// the panel will fit in nicely (355+45=400);
// X320x400
DIGITS 70,20,3,font01,1,health;
// displays the health and iq at 70,20 and
DIGITS 100,20,3,font01,1,iq; // 100,20 pixels from panel's upper left corner
// with 3 digits. Skills are multiplied by 1
// (no change) before they are displayed.
DIGITS 145,11,1,objfont,1,bluekeyplaced; // *placed skills are set to 0 a game
DIGITS 187,11,1,objfont,1,greenkeyplaced; // start, so these are their values
DIGITS 229,11,1,objfont,1,cakeplaced; // before picking up the objects.
DIGITS 271,11,1,objfont,1,fishplaced; // This way, the panel shows empty
// spaces (that is 0 in our objfonts)
FLAGS REFRESH; // big_panel is redrawn every frame cycle.
IF_KLICK place_objects; // if you pick up an object and click
// the panel, you will place it in the panel.
}

Let's say you pick up the cake. The skill cakeplaced takes the value 2 and will be displayed as cake.

And now, the last section. If we pick up an item, we need to change MSPRITE from the default mouse sprite (mouse.pcx) to cake, fish, keys. We need to use overlays:

OVLY greenkey, <objects.pcx>, 26,3,26,13
OVLY bluekey, <objects.pcx>, 56,3,33,15;
OVLY fish, <objects.pcx>, 92,4,33,13;
OVLY cake, <objects.pcx>, 93,19,33,14;
OVERLAY greenkey_ovl {OVLYS greenkey;}
OVERLAY bluekey_ovl {OVLYS bluekey;}
OVERLAY fish_ovl {OVLYS fish;}
OVERLAY cake_ovl {OVLYS cake;}

We have to know what overlay is active at any moment, so we define:

SKILL cursorgreenkey { VAL 0; }
SKILL cursorbluekey { VAL 0; }
SKILL cursorfish { VAL 0; }
SKILL cursorcake { VAL 0; }
SKILL cursornormal { VAL 1; } // MSPRITE = mouse_sprite at game start

We also need to know what objects are placed:

SKILL greenkeyplaced { VAL 0; } // nothing placed at game start
SKILL bluekeyplaced { VAL 0; }
SKILL fishplaced { VAL 0; }
SKILL cakeplaced { VAL 0; }

We shall define our objects as actors. They won't move, but we have two benefits:

  • We have 8 different skills for every actor that we can use;
  • Every actor can (and this time will) use overlays.

ACTOR greenkey_act {
TEXTURE greenkey_tex;
OVERLAY greenkey_ovl;
SKILL1 50; // SKILL1 holds the IQ value;
SKILL2 0; // SKILL2 holds the health value;
SKILL8 1; // SKILL8 will store the current MSPRITE;
}

You can use the other skills to add other proprieties to the player.
When you click the actor's texture, his corresponding IF_CLICK action
will be initiated. For all our actors, we use the same action:

ACTION takeobject {
IF_EQUAL cursornormal,0; // if we have picked an object we can't
END; // pick another one before placing the first
IF_EQUAL MY.SKILL8,1; // if actor's SKILL8 is 1, then we have picked
SET cursorgreenkey,1; // the green key and so on.
IF_EQUAL MY.SKILL8,2;
SET cursorcake,2;
IF_EQUAL MY.SKILL8,3;
SET cursorfish,3;
IF_EQUAL MY.SKILL8,4;
SET cursorbluekey,4;
SET MY.INVISIBLE,1; // after clicking the actor
SET MY.PASSABLE,1; // it becomes invisible
SET MSPRITE,MY.OVERLAY; // MSPRITE changes to actor's overlay
ADD iq,MY.SKILL1; // iq gets increased by actor's SKILL1
ADD health,MY.SKILL2; // health gets increased by actor's SKILL2
SET cursornormal,0; // we reset cursornormal
}

How do we place the objects? With:

ACTION place_objects {
SET greenkeyplaced,cursorgreenkey; // the skills displayed by the panel
SET cakeplaced,cursorcake; // are updated
SET fishplaced,cursorfish;
SET bluekeyplaced,cursorbluekey;
SET MSPRITE,mouse_sprite; // MSPRITE returns to pointer
SET cursornormal,1; // and we are allowed to pick other objects
}

That's all. You can find some more comments in our standalone demo.

Underwater regions and airbars:

Underwater regions are always looking good in games. With Acknex, they are easy to do, either. We will show you how to build them. Looking for an airbar code? You are at the right place.

First of all, we need to define two below regions, one above the other, to serve us as under water and out of water regions.

REGION belowroom01 { // that's our under the water region
FLOOR_HGT 0;
CEIL_HGT 45;
FLOOR_TEX floor_tex;
CEIL_TEX uwater_tex;
AMBIENT 0.4;
IF_ARISE region_arise; // this triggers an action in move.wdl
}

REGION room01 { // that's our out of the water region
FLOOR_HGT 45;
CEIL_HGT 275;
FLOOR_TEX water_tex;
CEIL_TEX ceil_tex;
AMBIENT 0.7;
BELOW belowroom01;
IF_DIVE region_dive; // triggers an action in move.wdl
}

Please note that first region's ceiling height must be equal with the second's floor height. And now, we need to define the airbar panel:

BMAP airbar_map,<airbar.pcx>;

PANEL airpanel {
VBAR 280,30,20,airbar_map,2,secspassed; // changes with secspassed
FLAGS REFRESH;
}

The skill secspassed has a maximum value of 10, so the player dies if he remains more than 9 seconds under the water.

When diving, the ACTION counteron is triggered in move.wdl:

ACTION counteron {
SET SECS,0; // reset the seconds
SET EACH_TICK.05,update_panel;
SET PANELS.10,airpanel; // shows the aibar panel
}

ACTION update_panel {
IF_ABOVE secspassed,9; // if more than 9 seconds have passed
BRANCH diejerome; // the player dies
RULE secspassed = SECS; // updating secspassed every second
}

ACTION diejerome { // we didn't want to kill the player in this demo
BEEP; // simply ADD health,-1000 if your player has health implemented
WAITT 32; // the beeps are delayed.
}

When arising, the ACTION counteroff is triggered in move.wdl:
ACTION counteroff {
SET secspassed,0; // resets secspassed
SET EACH_TICK.05,NULL;
SET PANELS.10,NULL; // hides the airpanel
}

Zoom in / out without errors:

Zoom in / out without errors: is this possible?

Zooming in and out is great. Most of the games aren't using a decent zooming. Except yours! Using a small trick (and Acknex) you will get a good progressive zooming, without any image errors! Just read on. 

The tutorial shows us how to do a normal zoom in / out. This had to be placed in move.wdl to get the zoom working:

SKILL PLAYER_ARC { MIN 0.5; MAX 2; }
IF_ABOVE KEY_INS,0;
RULE PLAYER_ARC = PLAYER_ARC - 0.1;
IF_ABOVE KEY_DEL,0;
RULE PLAYER_ARC = PLAYER_ARC + 0.1;

When I first tried it, I realized that I need to do two corrections:

  • PLAYER_ARC needs to be MAX 1 to get the proper zooming effect, because at game start, PLAYER_ARC = 1;
  • I wanted to lower MIN 0.5 in order to get a higher zooming effect, but the projection of the textures was distorted, although if I was rotating the player a little afterwards, the errors were corrected.

I finally got this working:

SKILL PLAYER_ARC { MIN 0.1; MAX 1; }
IF_ABOVE KEY_INS,0;
RULE PLAYER_ARC = PLAYER_ARC - 0.03;
IF_ABOVE KEY_INS,0;
ADD PLAYER_VROT,0.001; // easily rotate the player while zooming
IF_ABOVE KEY_DEL,0;
RULE PLAYER_ARC = PLAYER_ARC + 0.03;
IF_ABOVE KEY_DEL,0;
ADD PLAYER_VROT,-0.001;
}

Video-Sprites:

Creating BMP objects through digital image processing 

Bitmap objects with a two-dimensional appearance constitute one of the basic problems of creating a game - they need a lot of work and constitute a considerable source for errors even for gifted draughtsmen through the alternating views they present. Polygones on the other hand lack life-like curves. Lands of Lore 2: Götterdämmerung offers digitised video sequences of single sprites. This method may be applied to Acknex as a "3D Scanner" using a video camera, offering many varieties of usage. 

 

 

Required Hardware: 

Apart from a videocamera - any one using any one format will do - you'll need a video board that allows you to digitise analog videos. A powerful board with excelent performance I can for example recommend you Fast AV-Master (available at about $700); a PCI video board capable of processing the huge amounts of data generated by digital videos. Less powerful boards may cause a disrupted succesion of images due to the need to load the image. But if you wish to incorporate only videos that were substantially reduced in size, the use of a cheaper video board may be sufficient (ask your local dealer!). 

To install the board, depending on the board all you have to do is to reserve the specified memory area (in the Windows95 Hardware Settings). 

After you have installed the board you may plug a camera or VCR directly into the VHS or S-VHS inlets and archive the sequence to be digitised to the hard disk. The necessary transfer cables usually are supplied together with the video camera.  

Required Software: 

Basic image processing software is usually included when you buy a quality video board, but you will find it unsuitable for the purposes of effective sequencing. The currently best software for video processing is either Ulead-Studio 2.5 or Adobe Premiere 4.2. The latter offers a wealth of additional filters that may be applied to the single objects.

 

For postprocessing of bitmaps you have created an effective graphic program like Paint Shop Pro 4.0 will do.

 

 

Producing Sprites with the video camera:

The video camera can be used as a 3D scanner for characters, creatures or simply for objects. 

Example A: THING bottle (two dimensional) 

It is our intent to create a row of different bottles and glasses for a bar. So instead of sitting down and labouriously drawing the row bottle by bottle, realising that your talent for drawing with the mouse barely covers creating a game of Tic-Tac-Toe, we use the video camera. 

Put the object to be recorded in front of a neutral background (preferably green or blue). The camera records the object from the angel chosen by us - preferably one that corresponds to the point of view of our game character - and then we digitise our short video of a bottle to the hard disk.  

Using our video processing software we now export the frame of our choice into a graphics file.  

Afterwards we open the graphics file with the graphic software and adjust whatever needs adjusting: size, little errors... Then we select our bottle as the object to be copied and paste it in our graphic file for game objects. What remains is to include the graphic section into our WDL... 

Example B: THING Bottle (three dimensional) 

The technique may be put to further use for those bitmap fanatics who enjoy walking around objects. We again place the object in front of a neutral background, but this time on something like a potter's wheel. Then we record a slow(!), complete revolution of the object (changing the angle of the light at the same time, to avoid having the back of the object, supposed to lie in the shadow, appear as well lighted as the front). 

Again we digitise the sequence and call it up with the video processing software. We then shorten the sequence to the desired number of images. Which means that the movement needs to be slower than the number of sides of our intended object. 

But now we do not export a single frame, but create the entire revolution as a numbered row of bitmaps, using the software options. 

Afterwards we again switch to our graphic software, where we cut each single frame, format it and write it into a separate character graphic. 

And the rest is...the usual WDL programming 

Example C: ACTOR Fighter  

The same principle applies to persons; we record a person wearing the desired outfit in front of a neutral background, having him or her performing complete and self-contained movements. It is rather important under the circumstances that the movements of the different actions originate from the same starting conditions. 

We may for example define a fighter with the following movements: 

  1. Standing (Waiting)
  2. Turning around
  3. Walking (towards the player)
  4. Dialogue
  5. Fighting (against the player)
  6. Retreating A (away from player)
  7. Retreating B (towards a group of allies)
  8. Dying

The character of the fighter now offers potential for complex interaction (simple shot-'em-up á la Doom does grow boring after while, after all). The standing fighter (fucntion standing) may (but does not have to) be visible from all sides, for example allowing the player to sneak up to him from behind; so what we need is an actor turning around in the exact same posture in a sort of stop motion mode, so that we later can extract single images from our video. 

As soon as we "wake up" the fighter according to his paramters (e.g. IF_NEAR of a sentinel) the fighter turns around (function turning around - implying that we need at least eight different views of the start position) and starts moving towards us (function walking). We now need a complete walking motion. Attention: the arms must be in the same position at the beginning and the end of the walking cycle! Recording the actor from a somewhat greater distance, using a telephoto lense, does have the added advantage of avoiding a too hurried progress of the actor towards the camera which would create unfavourable changes of perspectives.  

The dialogue mode also only call for a frontal view of the fighter, who is bound to face the player each time he talks to him. The same is true for the fighting mode.  

The function Retreating A is also rather simple, as the fighter is moving away from us along a straight line; the same rules that were valid for the advancing movement  apply here. 

Opposed to that the function Retreat B is rather demanding, because we need a self-contained movement from at least 8 directions of movement performed smoothly, to enable us to "edit" to different perspectives (e.g when intercepting the retreating fighter). 

The function Dying may be well served with one position (preferably lying down), but does offer a more realistic view if comprised of different views.

 

 

Producing sprites using a video camera and morphing:

Making films and then bitmaps  out of people in costumes is something easily done. It is less easy when you're looking for other lifeforms, dragons, for example.  

But even here a video camera and morphing software (available on many shareware CD ROMs) make it a job easily done. 

Example D: THING/ ACTOR Dragon 

As a first step, we create a screenshot of a miniature dragon (very detailled tin models are available from role play specialty outlets!) and save it as a graphics file. 

We only want the dragon to be able to open his jaws and slightly lower his forelegs. 

We then smooth out the graphics file by exchanging the almost-neutral background we put on video by a onehundred percent single colour background. So here's ourstart image:

We then cut out hands and jaws separately and rotate the respective cut until it has reached the desired position, saving the outcome as a separate image.  

Then we retouch hand and lower jaw from our start image, insert the separate image elements hand and lower jaw, saving the result after applying the final touches as our final image.

 

Then we open both images in our morphing software (Ulead-Morph proves to be more convenient than Photomorph). Then we define in close succession the picture elements to be moved in the start image. The software locks onto the final image in the same X/Y position. Here we move the picture elements to the desired positions and make the software compute a sequence of bmps instead of an AVI.   

And last but not least we cut the singele images, change it to the correct size and paste it to the respective spot in the dragon's movements phases image file. 

This method may be used for THINGS and with greater expenditure of work, for AcTORS as well. Potential of this simplified F/X method may easily be extended to FLI/FLC film sequences in cut scenes.
