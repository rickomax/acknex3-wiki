How to write a beam-action?

Show me the Trekkie who doesn't know the words: "Beam me up Scottie!" And show me the D**M-Freak who doesn't expect to find beaming points in secret-regions. It's simply a standard. And certainly a must for every designer during development of a game, as  constantly shifting the player in the Wed does get on your nerves after a while. And even in the game itself there are few problems you can easily solve with a hidden beam action: for example if the player is to be placed on a higher or lower story that cannot and should not be created as belows for performance reasons. It thus does make sense to write yourself a little beam action and assign it to a certain key. Of course you should not forget to disable this "cheat" before the things go out into the stores....!
 

    1. Debug-Beam (simple jump)

To begin with, we assign the entire beam action to the key "b": 
    IF_B beam_me;
Afterwards we define ourselves a Thing serving as a mark for the jump. The Thing may have any texture whatsoever or even be invisible (Flag Invisible):
  
    THING Beamer      {TEXTURE transparent_black_tex; FLAGS INVISIBLE;}
Afterwards we beam the player to the region which should already be defined in the WED and contain the beam thing. At the same time the player is given the X and Y positions of the beam actor and the height of floor in the target region. And then the key "b" is set to the next beam action. This way we get a toggle. 
  
    ACTION beam_me {
     SET PLAYER_X, beamer.X; //Set Player X-Position of Actors
 SET PLAYER_Y, beamer.Y; // Set Player to Y-Position of Actors
 SET HERE, beamer.REGION; //Set Target Region (Region of Target Thing)
 RULE Player_Z = HERE.FLOOR_HGT + PLAYER_SIZE;  //Calculates height of player
 SET IF_B, beam_me2; //Set key B to new action
    }
And here again there are another beam actor, another target region etc.
  
    ACTOR Beamer2      {TEXTURE transparent_black_tex; FLAGS INVISIBLE;}
    ACTION beam_me2 {
     SET PLAYER_X, beamer2.X;
 SET PLAYER_Y, beamer2.Y;
 SET HERE, beamer2.REGION; //Set target region
 RULE Player_Z = HERE.FLOOR_HGT + PLAYER_SIZE;  //Calculates height of player
# SET IF_B, beam_me3; //another beam action, otional and left out here
    }

    2. Player-Beam (Player is tricked)

Now we are going to transport the player to another place on the map by adding or subtracting certain values to or from his position. Actions like that are best put to use when the player is not supposed to realise that he was beamed or if the topography demands such actions, e.g. when diving into an under water region that is not a below regions at the same time but does sport an entirely different topography below water (like ledges or grottoes). In such a case we transport the player into en entirely different region. The advantage you get from adding/substracting the values is obvious: if we assign the player a fixed position as a target like we did in the previous example, the movements may appear less than smooth if the player is beamend at the IF_ENTER of a region. Because how should we know at which point the player has entered the region or triggered an IF_DIVE? It doesn't matter when we are adding/subtracting. The player will always keep his apparent position.
In order to do that we start with finding out what the target region. Set any vertex in the start region, and another one in the target region. Connect the vertices and read the length of the wall that was created from the status line below. Lets assume that we need to take 1000 Steps in direction of y, east from the point of view of the zero point. So we need to add. For the sake of simplicity we should also make sure that the player is not moved diagonally across the map, but only in direction of x or y. The beam action itself we assign to an IF_ENTER of the start region. 
  
    REGION startregion {
        CEIL_HGT    0;
    FLOOR_HGT   12;
    CEIL_TEX    Decke_tex;
    FLOOR_TEX   boden_tex;
    IF_ENTER    beam_me_up_scottie;
}
     REGION zielregion {
    CEIL_HGT        0;
    FLOOR_HGT       12;
    CEIL_TEX        Decke_tex;
    FLOOR_TEX       boden_tex;
   }
     And now the beam action::
      
    ACTION beam_me_up_scottie {
     ADD PLAYER_Y, 1000; //Adding the beaming value to the Y position of the Actor
 SET HERE, Zielregion; //Set target region 
 RULE Player_Z = HERE.FLOOR_HGT + PLAYER_SIZE;  //Height of player is calculated
 }
 
 
    3. Finding the region/Synonyms
Another little specialty to close this part: the good old Skaphander had a service level the player could beam himself to at any time using the ESC button, in order to save scores, change options or just rest a little. Pressing the ESC button one more time beamed him right back to the very spot in the level he left when he beamed himself into the service level. But how do you keep in mind the respective region?

You can use a synonym to achieve this elegantly with a single line. But what is a "SYNONYM"? A synonym is something standing in for something else. Our WDL allows us to define different types of synonyms. A few are pre-set: HERE, THERE (and everywhere..),MY, etc.

But we are allowed to define as many synonyms as we please. And here's how:
  

    SYNONYM  mein_herzschrittmacher {TYPE ACTOR;}
The following TYPE's are allowed:  OVERLAY, TEXTURE, WALL, THING, ACTOR, REGION, PANEL, TEXT, STRING oder ACTION.

In our case we are going to need a synonym of the REGION type, because we want to adress a region. And so the line goes: 
  

    SYNONYM    LastRegion {TYPE    REGION;}
Of course we also need to remember the old x and y positions of the player. This we do with simple skills: 
  
    SKILL Last_x {}
    SKILL Last_y {}
    SKILL Last_z {}
And the beam-action: 
    ACTOR Beamer3      {TEXTURE transparent_black_tex; FLAGS INVISIBLE;}
    ACTION beam_me3 {
     SET Last_X, Player_x;        //this is where we remember the old positions
 SET Last_Y, Player_y;
 SET Last_Z, Player_Z;
     SET LastRegion, HERE;        //and here we remember the old region!
 SET PLAYER_X, beamer3.X;
 SET PLAYER_Y, beamer3.Y;
 SET HERE, Beamer3.REGION;
 RULE Player_Z = HERE.FLOOR_HGT + PLAYER_SIZE;  //height of player is calculated

 SET IF_ESC, beam_me_back;
    } 

ACTION beam_me_back {
     SET Player_X, Last_X;        //here we set the old positions
 SET Player_Y, Last_Y;
 SET Player_Z, Last_Z;
 SET HERE,LastRegion;        //the whole thing backwards!
 SET IF_ESC, beam_me3;
    }
     
There is another way to assign the player a new region: the command LOCATE, which is simply written into the action as a KEYWORD:
  
    ACTION beam_me_back {
     SET Player_X, Last_X;        //here we set the old positions
 SET Player_Y, Last_Y;
 SET Player_Z, Last_Z;
 SET IF_ESC, beam_me3;
     LOCATE;
    }
     
This command makes sure that the player will find his way back to his region. But it does use up a lot of computing time, depending on the size of the WMP! That's why we use this command only when loading a completely new level.

Markus Fabian: Textures from the Faller Archives!

Here is a simple but quite useful tip from one of our readers: whoever's fed up with browsing through cheap CDs, looking for textures fit for walls, floors, ceilings etc. only to find out that the damn things aren't tiled and need to be edited with DPaint or PSP anyhow, is advised to contact his nearest toy shop carrying toy trains. Because for years a company called Faller has been offering cardboards with wall textures sized 25x12.5 cm to DIY Railway fans. And these textures (stone, brick, clinker) are tiled! So all you need to do is scan them and adjust them to the respective palette.  
There is bound to be an equivalent available to our American friends (who knows, maybe Faller is even selling in the USA). The cardboards tested by us are called:
 
      HO 605 (Clinker)
      HO 610 (Quarry)
      HO 607 (Industrial bricks)
       

      To those who do not own a scanner: many photo shops offer that service! And you can order from Faller under the following number: 07723 - 6510.
       
       
       

 
 

Of Mice and Men

 

No mouse: no clicks. An adventure without clicks is not an adventure. And 3D Games usually don't include click actions. But that is not so with ACKNEX, which is supposed to work as an action- as well as an adventure-engine. In order to bring a mouse pointer to the screen, we need to draw it first. The file we then give a name like "arrow.pcx" and assign it to the overlay:
  

    OVLY        mouse_ovr,<arrow.pcx>;
    OVERLAY     mouse_sprite { OVLYS mouse_ovr; }
    MSPRITE     mouse_sprite;
That's all! And now: the action activating the mouse: 


ACTION mouse_toggle {
     ADD   MOUSE_MODE,    2;    //in Mode 1 the mouse influences player movement. 
 IF_ABOVE MOUSE_MODE, 2;
 SET MOUSE_MODE,      0;
 IF_EQUAL MOUSE_MODE, 0;
    END;
    loop:
     WAIT   1;
 RULE  MOUSE_X = MOUSE_X + 2*MICKEY_X;  //requesting the current mouse position.
 RULE  MOUSE_Y = MOUSE_Y + 2*MICKEY_Y;
 IF_ABOVE MOUSE_MODE,0;
    GOTO loop;
    }
And that's all there is to it. If you want the mouse to display something when touching upon a certain point in the level, we still need textures with TOUCH, IF_KLICK etc. Quite simple, that one.

To define a texture with touch we proceed as follows:
To begin with a FONT, because nobody can write without letters:
  

    FONT    show_font, <show.pcx>,12,12;
where the parameter following the assigning of the file give the width and height of the single letters in pixels. Do not forget:
the bitmap must either contain  11, 128, or 256  characters (because of the proportions). If that is not the case, you'll get an ugly error message!

And then a string, 'cause what else are we supposed to read? 

    STRING    screwdriver_str, "This is a screwdriver!";
And now, finally, the definition! 
    TEXTURE show_me {
    SCALE_XY   16,16;
    BMAPS      yep;
    FONT       show_font;  // To put out a string, we need a font!
    TOUCH      Schraubenzieher_str;  //  Even this string t´need to be defined somewhere!
    }
This Texture we assign to a Thing or an Actor. Each time our mouse pointer slides across the central pixels of this Thing or Actor, the text "This is a screwdriver!" appears. Hopefully!

And now add an IF_KLICK. Assigning it is as easy as making a cake (even though nobody seems to be able to do that properly!). Simply define a TEXTURE with an IF_KLICK:
  

        TEXTURE     Tuer_auf_tex {
        SCALE_XY        16,16;
        BMAPS           tuer_tex;
        IF_KLICK            aufmach;
}
And that's all! The   ACTION "Aufmach" ( means: open up)  should be there as well of course. It doesn't matter whether the actions triggers a shot, paints the player blue of returns the string "Gotcha!", but an ACTION it must be! !
Have fun!
